# NexGenX Windows Agent — Deployment Model

## The Core Question

When a customer downloads our installer onto their Mac/PC, where does the
actual code come from? Can we host it on GitHub Free?

## Answer: Yes — Use GitHub Free for Distribution

GitHub Free gives you:
- Private repos (unlimited, you have this already)
- Releases with binary downloads (no limit on free)
- Raw file hosting for installer scripts
- CDN for downloads (fast worldwide)

This is **exactly the same model** used by:
- Microsoft (PSScriptAnalyzer, PSReadLine)
- HashiCorp (Terraform, Vault, Packer — all on GitHub Free)
- Tailscale (their client is open source on GitHub)
- Most open-source companies

**You don't need GitHub Enterprise, paid GitLab, or your own artifact server.**

## The Two Different Things We're Distributing

There are two pieces that flow from us to the customer:

### 1. Installer Script (small, ~2 KB PowerShell)
- This is a **bootstrap** — it doesn't contain the app
- It just connects to GitHub and pulls the actual binaries
- Hosted as a raw file: `https://raw.githubusercontent.com/NexGenX/ngx-windows-agent/main/install.ps1`
- Customer runs: `irm https://raw.githubusercontent.com/NexGenX/ngx-windows-agent/main/install.ps1 | iex`
- (One line — no file download needed)

### 2. The App Itself (Python source, ~50 KB total)
- Hosted in a **GitHub Release** as a zip / tarball
- Or distributed via **raw files** in the repo
- The installer pulls from `https://github.com/NexGenX/ngx-windows-agent/releases/latest`

## Recommended Setup: GitHub Releases

For each version, we publish a release. The release page provides:
- `ngx-windows-agent-v1.0.0.zip` — bundled installer
- Source code (auto-generated by GitHub)
- Release notes / changelog
- Checksums for verification

Customer workflow:
```
1. Goes to https://github.com/NexGenX/ngx-windows-agent/releases/latest
2. Downloads the latest release zip
3. Runs the included install.ps1 as Administrator
4. Done — agent is installed and running
```

OR (one-liner via PowerShell):
```
irm https://github.com/NexGenX/ngx-windows-agent/releases/latest/download/install.ps1 | iex
```

The `latest/download` URL **always points to the newest release** — no version number in the URL.

## Security Considerations for GitHub-Distribution

Because the script runs as Administrator, we need to:

1. **Sign releases with checksums** — Provide a SHA-256 file alongside
   the release. Customer should verify before running as admin.

2. **Use HTTPS only** — GitHub forces HTTPS, so we're good.

3. **Pin dependencies** — Don't fetch random stuff from the internet
   during install. All deps are `pip install` from PyPI (which is also
   HTTPS by default).

4. **Never commit secrets** — `.gitignore` already blocks `.env`,
   `*.key`, `*.pem`, `access_codes.txt`. The customer's access code is
   generated locally on first run, never transmitted to GitHub.

5. **Consider signing releases with GPG** — GitHub shows a "Verified"
   badge if you sign the commit/tag. Adds trust.

## Alternative: Self-Hosted Distribution (Not Recommended Yet)

If you ever outgrow GitHub Free (unlikely), you can:
- Self-host on a server in your datacenter (S3-compatible, like MinIO)
- Use Azure Blob Storage ($0.018/GB/month — pennies)
- Use Cloudflare R2 (10 GB free, $0.015/GB after)

**For v1, GitHub is the right choice.** Don't pay for hosting you don't need.

## The Installer Script's Behavior (Updated)

When the customer runs the installer, it should:

1. Download the latest release from GitHub
2. Verify the SHA-256 checksum (customer prompted if mismatch)
3. Extract to `C:\NexGenX`
4. Install Python deps via pip
5. Generate the access code
6. Create the scheduled task
7. Open firewall for port 9400
8. Show the access code in a notification

**No external calls except to GitHub and PyPI.** No telemetry, no
phone-home (in v1.0).

## Update Flow

When you push a new version:
```bash
git tag v1.1
git push --tags
gh release create v1.1 --generate-notes
```

Customers can update via:
- The tray app menu (with your approval of update URL)
- Manual: re-run the installer — it overwrites the existing install
- Or: an auto-update mechanism (v1.1 feature)

## What About Mac / Linux?

You mentioned Mac earlier. The same architecture works:
- **Mac agent** uses AppleScript / Quartz instead of uiautomation
- **Linux agent** uses X11 / AT-SPI instead of uiautomation
- All three (Win/Mac/Linux) talk the same REST API
- One Linux LXC gateway can control VMs of any OS

We'd write a parallel `agent_server_mac.py` and `agent_server_linux.py`
that implement the same endpoints, but use the native APIs for input
control and screen capture.

Same GitHub repo, just additional files:
```
ngx-windows-agent/
├── agent_server.py        (Windows)
├── agent_server_mac.py    (macOS)
├── agent_server_linux.py  (Linux)
├── install.ps1            (Windows installer)
├── install_mac.sh         (macOS installer)
├── install_linux.sh       (Linux installer)
└── ...
```
